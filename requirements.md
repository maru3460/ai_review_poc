# PR構造可視化レビュー支援ツール 要件定義書（統合版 PoC v1.0）

## 1. プロジェクト概要

### 1.1 目的
PR（Pull Request）の差分と既存コード文脈をAIで解析し、変更内容を複数の視点で構造的に可視化する。
レビュワーが「差分の行単位確認」だけでなく「影響・意図・設計整合性」を短時間で把握できる状態を実現する。

### 1.2 コンセプト
- 読むレビューから、見るレビューへ
- PRを「差分」ではなく「構造変化」として扱う
- 同じPRをモード切り替えで多面的に理解できる

### 1.3 ターゲットユーザー
- 大規模PRを扱うエンジニア
- 他チームのコードレビュー担当者
- プロジェクト新規参加者（オンボーディング用途）

## 2. スコープ

### 2.1 PoC対象
- GitHub PR（単一リポジトリ）
- JavaScript / TypeScript / Ruby / C#
- Webアプリ開発文脈（例: React + Node.js）

### 2.2 PoC非対象
- 上記4言語以外への対応
- 完全精度の静的解析
- 本番運用レベルの性能最適化
- CIへの深い統合（必須要件外）

## 3. ユースケース
- 修正ファイルが多いPRで、主要ロジック変更を素早く掴む
- 変更関数の影響範囲を追跡し、想定外の波及を検知する
- PR説明と実装内容の整合性を確認する
- 設計ルール違反（レイヤ越境など）の早期検出に使う

## 4. システム全体像（PoC）
1. PR作成/更新WebhookまたはBot mentionをトリガーに解析ジョブ開始
2. GitHub APIでPR差分と関連メタ情報を取得
3. AST/LSPでシンボル・依存情報を抽出
4. 依存グラフ/呼び出しグラフを生成
5. LLMへ「差分 + 関連コード + メタ情報」を入力
6. モード別の構造化出力（JSON/Mermaid）を生成
7. 可視化ページを生成し、PRへURLをコメント投稿

## 5. 機能要件

### 5.1 PR連携・起動
- PR作成/更新時にBotが可視化ページを生成し、PRコメントへURLを自動投稿できる
- PR上でBotをmentionした場合、オンデマンドで可視化ページを生成し、URLをコメント返却できる
- 変更ファイル一覧、追加/削除行、変更関数一覧を取得できる

### 5.2 静的解析（PoCレベル）
- 抽出対象: 関数定義、クラス、import/export、関数呼び出し
- 出力対象:
  - ノード: file/function/class
  - エッジ: call/import/inherit

### 5.3 モード切替UI
- タブでモード切替できる
- 切替時に同一PRの別視点を即時表示する
- ノードクリックで該当コード断片とAI解説を表示する

## 6. 可視化モード要件

### 6.1 Workflow Changeモード
- 目的: 処理フローの前後差分を把握
- 表示: フローチャートまたはシーケンス図（前後比較）
- 強調: 追加/削除/修正ステップ、分岐変化
- PoC制約: 呼び出しツリー展開は最大3階層

### 6.2 Impact Mapモード
- 目的: 影響範囲とリスク箇所の把握
- 表示: 変更ノード起点の依存ネットワーク/波紋表現
- 強調: 影響距離、関係強度、高リスクノード

### 6.3 Data Lineageモード
- 目的: データの流れと副作用の把握
- 表示: 入力（API/Request）から出力（DB/UI）までの変遷
- 強調: 変数/オブジェクトの加工経路、副作用有無

### 6.4 Architecture Complianceモード
- 目的: 設計整合性の確認
- 表示: レイヤ構造上に変更をマッピング
- 強調: レイヤ越境依存、逆方向依存、新規依存

### 6.5 Intent/Contextモード
- 目的: PR意図と実装整合の確認
- 入力: PRタイトル、PR説明、差分要約、変更関数一覧
- 出力: 意図分類、主目的、関係の薄い変更、懸念ポイント
- 表示: 要約パネル + 意図ツリー/注釈

## 7. 画面要件
- 左: 変更ファイルツリー
- 中央: モード別可視化キャンバス（ズーム/パン対応）
- 右: AIによる3行要約とレビュー観点、リスク注釈

## 8. 非機能要件
- パフォーマンス:
  - 小規模リポジトリ（~500ファイル）で初期表示10秒以内
  - 典型的なPRで主要可視化30秒以内
- 精度:
  - PoC段階では静的解析70%程度の実用精度を許容
- UX:
  - インタラクティブ探索（クリック・ハイライト・コードジャンプ）

## 9. 技術スタック（PoC想定）
- Backend: Node.js, GitHub API, tree-sitter/ts-morph/LSP
- LLM: GPT-4o/Claude/Gemini系（抽象化レイヤ経由で差し替え可能）
- Frontend: React, React Flow or D3.js/Cytoscape.js, Mermaid.js
- 連携: GitHub App（将来はGitLabも検討）

## 10. データモデル（最小）
```ts
Node {
  id: string
  type: 'file' | 'function' | 'class'
  module: string
  changeType: 'added' | 'removed' | 'modified' | 'unchanged'
}

Edge {
  from: string
  to: string
  type: 'call' | 'import' | 'inherit'
}
```

## 11. 成功基準（PoC）
- モード切替で「見え方の違い」が明確に体感できる
- 図解のみでPR意図/影響の8割を事前把握できる
- レビュワーが有用性を実感できる（定性フィードバック）

## 12. 今後の拡張

- CI統合、VSCode拡張
